# システム要件定義書

## システム概要
認知機能向上ツール「CogEvo」を利用する介護・福祉施設スタッフ向けの問い合わせ対応AIチャットボット。Gemini APIを活用し、既存のPDFマニュアルやFAQを基に24時間365日自動応答する。ITリテラシーが高くないユーザー層に配慮し、共通パスワードによる簡易認証、音声入力対応、平易な言葉への翻訳機能を備え、解決できない場合は問い合わせフォームへスムーズに誘導する。

1. システムの目的
   - 問い合わせ対応工数の削減と社内エキスパートの負担軽減
   - 施設スタッフの待ち時間解消と自己解決率の向上
   - 顧客満足度の向上

2. 主な機能
   - **コア機能**: テキスト/音声/クイック質問によるチャット、共通パスワード認証、問い合わせフォーム誘導
   - **差別化機能**: エンジニアの知識をメンターの語り口で翻訳する応答生成
   - **付加価値機能**: ブラウザ標準機能を用いたコストゼロの音声入力

3. 提供する価値
   - **ユーザー**: 専門用語に悩まされず、その場でトラブルを解決できる
   - **ビジネス**: サポートコストの削減と、本来の業務へのリソース集中

4. 実現方法の概要
   - **Gemini API**: 高精度な回答生成とコンテキスト理解
   - **RAG (Retrieval-Augmented Generation)**: 社内ドキュメントに基づく正確な回答
   - **Web Speech API**: ブラウザネイティブの音声認識

## 情報の前提と仮定
- **情報源**: `/docs/input` フォルダ内のビジネス要件、プロダクト要件等
- **仮定**:
  - ユーザーは一般的なWebブラウザ（Chrome/Safari）を利用可能である
  - 初期ユーザー数は数百施設規模、同時接続は数十程度と想定
  - 既存ドキュメント（PDF/FAQ）はテキスト抽出可能な形式である

## 機能要件

### 1. ユーザーインターフェース (フロントエンド)
- **認証画面**: 共通パスワード入力のみのシンプル設計
- **チャット画面**:
  - 対話形式の表示（ユーザー質問/AI回答）
  - テキスト入力欄と送信ボタン
  - 音声入力ボタン（マイクアイコン、録音中インジケータ）
  - クイック質問ボタン（頻出質問のワンタップ送信）
  - 免責事項の表示
- **問い合わせフォーム**:
  - 施設名、担当者名、連絡先、内容の入力
  - 送信完了画面

### 2. バックエンド / AI処理
- **認証処理**: 共通パスワードの検証
- **検索・回答生成 (RAG)**:
  - ユーザー質問に関連するドキュメントの検索
  - Gemini APIへのコンテキスト注入と回答生成
  - 「メンターの語り口」へのスタイル調整
- **フォーム送信処理**:
  - 入力内容のバリデーション
  - サポート担当者へのメール通知

### 3. データ管理
- **ドキュメント管理**: PDF/テキスト/HP情報のベクトル化と保存
- **ログ管理**: 問い合わせ内容と回答の保存（改善分析用）

## 非機能要件
- **パフォーマンス**:
  - チャット回答の生成：平均5秒以内を目指す（UX向上のため「考え中」表示必須）
  - ページ読み込み：スマホ/タブレット回線でもストレスのない速度（LCP 2.5秒以内）
- **セキュリティ**:
  - 社外秘データは学習させない（物理的分離）
  - 通信の暗号化 (HTTPS)
  - 共通パスワードによる最低限のアクセス制限
- **ユーザビリティ**:
  - 高齢者施設スタッフ向けの大きな文字サイズとコントラスト
  - タブレット操作に適したタップ領域の確保
- **対応ブラウザ**: Chrome, Safari (iOS/iPadOS含む), Edge

## 技術スタック
今回は開発リソースが1名であり、迅速なMVP開発と将来の拡張性を考慮し、以下のモダンな構成を推奨します。

- **フロントエンド**: **Next.js (App Router)**
  - 理由: Reactベースの高速なUI構築、Vercelとの親和性、SEO/パフォーマンス有利
- **言語**: **TypeScript**
  - 理由: 型安全性による保守性の向上、開発効率化
- **バックエンド/API**: **Next.js API Routes** (Server Actions)
  - 理由: フロントエンドと同一プロジェクトで完結でき、管理コストが低い
- **AI/LLM**: **Gemini API (Google)**
  - 理由: 高速・安価、長文コンテキストに強い
- **ベクトルデータベース**: **Supabase (pgvector)**
  - 理由: PostgreSQLベースでRDBとベクトル検索を統合管理可能、低コスト
- **認証**: **カスタム実装 (環境変数での共通パスワード照合)**
  - 理由: 簡易認証（ID不要）のため、Clerk等のSaaSはオーバースペック。シンプルに実装する。
- **ホスティング**: **Vercel**
  - 理由: Next.jsの最適化、デプロイの容易さ、初期コスト無料

## 開発ロードマップ

### Phase 1: MVP開発 (1-1.5ヶ月)
- **基盤構築**: Next.jsプロジェクト作成、Supabase連携
- **RAG実装**: ドキュメントのベクトル化、検索ロジック実装、Gemini接続
- **UI実装**: チャット画面、音声入力(Web Speech API)、簡易認証
- **フォーム実装**: 問い合わせフォームとメール通知
- **社内テスト**: 回答精度の検証とチューニング

### Phase 2: 限定公開・改善 (0.5ヶ月)
- **限定公開**: 一部施設/特約店への展開
- **ログ分析**: 誤回答の修正、FAQデータの拡充

### Phase 3: 全体公開
- **製品リンク設置**: 一般ユーザーへの開放
- **運用フロー確立**: 定期的なドキュメント更新手順の整備

## リスク分析

### 技術的リスク
- **回答精度**: 専門用語の誤解釈やハルシネーション
  - *対策*: 参照ドキュメントの質向上、プロンプトでの「わからなければ誘導」の徹底
- **音声認識精度**: 施設内の雑音や方言による認識不良
  - *対策*: キーボード入力併用、誤認識時の修正UI

### ビジネス・運用的リスク
- **ドキュメントの陳腐化**: マニュアル更新時にAI知識が古くなる
  - *対策*: マニュアル更新時のベクトルデータ更新フローを業務に組み込む
- **パスワード流出**: 共通パスワードが部外者に漏れる
  - *対策*: 定期的な変更、または被害が少ない情報のみを扱う（個人情報等は扱わない）

## UI/UXの方向性
- **デザインコンセプト**: 「優しさ」「明快さ」
- **トーン＆マナー**: CogEvoブルーを基調とした、清潔感と安心感のあるデザイン
- **ターゲットデバイス**: タブレット（iPad等）ファースト、スマホ対応
- **参考イメージ**:
  - チャット部分: LINEのような親しみやすい吹き出し
  - ボタン: 大きく押しやすい、角丸のデザイン

## 追加情報リクエスト
1. **既存ドキュメントの量と形式**: PDFのページ数やファイル構造（テキスト抽出の難易度見積もりのため） - *優先度: 高*
2. **問い合わせフォームの通知先**: 具体的なメールアドレスまたは管理ツール - *優先度: 低*
3. **具体的なFAQデータ**: 現状のリスト（Excel/Text）の共有 - *優先度: 中*
